# Выкладка бинаря на узел

`deploy-binary.sh` — процедура ручного пуша собранного бинаря, с бэкапом и
ротацией.

## Зачем отдельный скрипт

`install-client.sh` / `install-server.sh` ставят релиз из тарболла и сами
делают бэкап с ротацией. Но обычная рабочая выкладка выглядит иначе — собрали
локально, положили на узел через `scp`, подменили, перезапустили — и мимо этих
скриптов проходит всё: и бэкап, и ротация, и проверка, что сервис после
рестарта действительно обслуживает.

Чем это кончается: к 2026-07-23 на семи узлах лежало **112 бэкапов, 1.6 ГБ**,
в трёх разных схемах именования, включая июньские. Плюс каждый пуш вручную
повторял одни и те же шаги, и любой можно было пропустить.

## Использование

```bash
ops/deploy/deploy-binary.sh <host> <unit> <local-binary> [keep]
```

```bash
# клиент на x86_64-узел
ops/deploy/deploy-binary.sh sysadm@cloud3.beerloga.su outline-ws-rust \
  target/x86_64-unknown-linux-musl/release/outline-ws-rust

# сервер на ARM-узел, оставить 5 бэкапов вместо 3
ops/deploy/deploy-binary.sh mmv@198.18.1.104 outline-ss-rust \
  target/aarch64-unknown-linux-musl/release/outline-ss-rust 5
```

Порт метрик выводится из имени юнита: `ws` → 9091, `ss` → 9090.

## Что делает и от чего страхует

1. **Отказывает при несовпадении архитектуры.** Парк смешанный — `.104`
   aarch64, остальные x86_64; перепутать цели легко, особенно когда собираешь
   обе подряд.
2. **Сверяет sha256 после передачи** — приехали ровно те байты, что собраны.
3. **Делает бэкап** `<bin>.bak.YYYYMMDDHHMMSS` через `cp -a` (режим и владелец
   сохраняются, иначе откат вернул бы файл с правами root'а).
4. **Ротирует по mtime**, оставляя `keep` последних (по умолчанию 3).
   Сортировка именно по времени, а не по имени: на парке соседствуют
   `.bak.20260716125448` и `.bak.pre-catA`, и по алфавиту свежий `pre-catA`
   идёт ПЕРЕД старым `reno-remnant` — то есть алфавитная ротация удалила бы
   актуальный откат. Проверено: на реальном наборе имён старая логика удаляла
   самый свежий бэкап.
5. **Проверяет, что сервис обслуживает** — `is-active` плюс `/metrics` = 200,
   до 30 секунд ожидания. Systemd считает юнит живым и когда рантайм завис
   (см. `ops/watchdog/`), поэтому одного `is-active` мало.
6. **Откатывается сам**, если после рестарта метрики не отдаются: возвращает
   только что снятый бэкап, перезапускает и выходит с ошибкой.

## Проверено

Прогнано на cloud4 (резерв) 2026-07-23:

- ARM-бинарь на x86_64-узел → отказ, `exit=1`, ничего не тронуто;
- обычная выкладка → бэкап снят, `/metrics` 200, sha совпал;
- 7 бэкапов при `keep=3` → удалено 4, остались три свежих по mtime, включая
  `pre-catA`; выброшен `zzz-old-but-late-alphabet`, который алфавитная
  сортировка сохранила бы.
