# Liveness-watchdog для outline-юнитов

Внешний сторож, который перезапускает `outline-ws-rust` / `outline-ss-rust`,
когда процесс жив, а обслуживание встало.

## Зачем, если есть systemd

systemd следит за **процессом**, а не за тем, обслуживает ли он кого-нибудь.
Отказ, ради которого это написано (cloud3, 2026-07-22, 3.5 часа простоя
незамеченными):

- юнит `active`, порты в `LISTEN`, `NRestarts=0`;
- cgroup душит процесс через `MemoryHigh` (`memory.events: high 714194`),
  единственный worker на 1 vCPU висит в `D` на `mem_cgroup_handle_over_high`;
- `oom_kill=0` — OOM-killer не сработал, значит systemd ничего не заметил;
- async-рантайм стоит целиком: `Recv-Q` на `:9091` растёт, скрейпы копятся,
  журнал молчит часами.

Ни `Restart=on-failure`, ни `WatchdogSec=` тут не помогают: первый ждёт
падения процесса, второй требует `sd_notify` из приложения. Единственный
сигнал, совпадающий с тем, что видит пользователь, — **отвечает ли юнит на
свой собственный `/metrics`**. Его и проверяем.

## Как работает

`outline-watchdog <unit> [url] [threshold]`:

- URL по умолчанию выводится из имени юнита: `outline-ws-rust` → `:9091`,
  `outline-ss-rust` → `:9090` (`/metrics` на `127.0.0.1`);
- успех обнуляет счётчик промахов в `/run/outline-watchdog/<unit>.misses`,
  промах инкрементит его и пишет WARNING в журнал;
- на `threshold`-м промахе подряд (по умолчанию 3) — `systemctl restart`;
- остановленный юнит (`systemctl is-active` = false) не трогается вообще,
  чтобы сторож не воевал с ручным `systemctl stop` во время обслуживания;
- счётчик обнуляется **до** рестарта: если рестарт сам подвиснет, следующий
  тик начнёт счёт заново, а не ударит повторно немедленно.

Таймер: `OnBootSec=2min` (даёт юниту подняться после перезагрузки) плюс
`OnUnitActiveSec=1min`. Итого от зависания до рестарта — максимум ~3 минуты.

## Установка

```bash
scp -r ops/watchdog <host>:/tmp/ && ssh <host> 'cd /tmp/watchdog && sudo ./install.sh'
```

Скрипт идемпотентен и включает таймер только для тех юнитов, что есть на
хосте. Стоит на cloud3, cloud4, `198.18.1.104`, `198.18.1.105` — по два
инстанса (ws + ss) на узел.

## Проверка

```bash
# здоровый путь: тихо, exit 0
sudo /usr/local/sbin/outline-watchdog outline-ws-rust; echo $?

# счёт промахов без риска рестарта (порог заведомо недостижим)
sudo /usr/local/sbin/outline-watchdog outline-ws-rust http://127.0.0.1:9/metrics 99

# что он делал
journalctl -t outline-watchdog
systemctl list-timers 'outline-watchdog@*'
```

Полное срабатывание (три промаха → рестарт) проверено вживую на cloud4:
`pid 844928 → 903549`, после рестарта метрики снова отдают 200, аплинки
зелёные.

## Отключить

```bash
sudo systemctl disable --now outline-watchdog@outline-ws-rust.timer \
                             outline-watchdog@outline-ss-rust.timer
```

## Что это не лечит

Сторож — обвязка вокруг симптома. Первопричина на маленьких узлах —
рост памяти под всплеском флоу и cgroup-throttle; лимиты и zram живут
отдельно (см. историю тюнинга парка). Каждый рестарт сторожа стоит
разбирательства: если он срабатывает регулярно, чинить надо память, а не
интервал таймера.
