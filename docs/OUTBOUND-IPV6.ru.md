# Выбор исходящего IPv6-адреса (source)

*English version: [OUTBOUND-IPV6.md](OUTBOUND-IPV6.md)*

Как `outline-proxy` выбирает **исходящий** IPv6-адрес для трафика, который он
отправляет в реальный интернет — и для upstream-соединений сервера
(`outline-ss-rust`), и для direct-дайлов клиента (`outline-ws-rust`, путь
`via = "direct"` / обход через TUN). Обычно цель — одно из двух
противоположных желаний, и они конфликтуют:

- **Стабильность** — держать долгие соединения (SSH, тайлы карт, стриминг)
  живыми, пока хост ротирует IPv6-адреса privacy-extensions.
- **Ротация** — разносить исходящий трафик по множеству source-адресов, чтобы
  наблюдатель не мог тривиально связать его весь с одним адресом.

## Чем ограничивает обратный трафик

Выбрать source — простая половина. Сложная — сделать так, чтобы **ответ**
вернулся. Что возможно, решают три факта:

1. **NDP.** На общем SLAAC-сегменте вышестоящий роутер шлёт Neighbor
   Solicitation на каждый адрес назначения и пересылает ответ, только если
   кто-то за этот адрес ответил. Ядро отвечает лишь за адреса, реально
   сконфигурированные на интерфейсе. Поэтому *случайный* (не сконфигурированный)
   source-адрес не получит ответ, если NDP за него никто не проксирует.
2. **Маршрутизация.** Если весь префикс **маршрутизируется** на хост
   (делегирование DHCPv6-PD или ручной AnyIP-маршрут), любой адрес из него
   возвращается без NDP-плясок.
3. **Анти-спуфинг (RPF / BCP38).** Вышестоящий узел может дропать пакеты,
   source которых «не ваш». Домашний роутер (например Keenetic) обычно нет;
   провайдер на общем сегменте — может.

`IPV6_FREEBIND` позволяет сокету **забиндить** нелокальный адрес, но никак не
помогает с пунктами (1)–(3) — это факты сетевого уровня, которые надо
обеспечить отдельно.

## Серверные режимы (`[outbound]` / `--outbound-ipv6-*`)

Можно задать ровно один режим — они взаимоисключающи.

| Режим | Ключ конфига | Пул source | Что нужно для обратного трафика |
|-------|--------------|------------|----------------------------------|
| Статический префикс | `ipv6_prefix = "2a01:…::/64"` | весь CIDR (random на коннект) | `IPV6_FREEBIND` + префикс on-link **или** AnyIP-маршрут; весь /64 routed/NDP-отвечаем |
| Пул интерфейса | `ipv6_interface = "enp1s0"` | адреса, реально на интерфейсе | ничего сверх — ядро владеет и отвечает NDP (работает под обычным SLAAC) |
| Динамический префикс | `ipv6_prefix_interface = "enp1s0"` | весь текущий /64 интерфейса, re-derive на refresh | то же, что статический префикс (NDP-proxy / ndppd), но /64 определяется, не зашит |

### `outbound_ipv6_prefix` — статический префикс
Биндит случайный адрес из фиксированного CIDR. Когда у вас есть routed-префикс,
который не меняется. Нужно, чтобы префикс был достижим обратно на хост (on-link
+ `IPV6_FREEBIND`, ставится автоматически, либо AnyIP-маршрут
`ip -6 route add local <prefix> dev lo`).

### `outbound_ipv6_interface` — пул адресов интерфейса
Перечисляет глобальные адреса, сейчас сконфигурированные на интерфейсе, и
биндит случайный. Так как это реальные адреса, ядро отвечает за их NDP и RPF
проходит — это **единственный режим, работающий на обычном SLAAC-сегменте без
ndppd**. С privacy-extensions (`use_tempaddr=2`) пул отслеживает ротируемые
temporary-адреса. На Linux пул читается из `/proc/net/if_inet6`, и
**deprecated / tentative / dadfailed адреса отсеиваются**, поэтому
вот-вот-удаляемый temporary не пиннится (иначе flow порвался бы на ротации).

### `outbound_ipv6_prefix_interface` — динамический префикс из интерфейса
Золотая середина: определяет текущий глобальный **/64** интерфейса и биндит
случайный адрес по *всему* префиксу (как статический), но **re-derive /64 на
каждом refresh** (`outbound_ipv6_refresh_secs`). Когда префикс динамический —
например провайдер делегирует префикс, меняющийся при каждом реконнекте, а
роутер (Keenetic) ре-анонсирует его в LAN по SLAAC. Как и статическому
префиксу, нужен весь /64, маршрутизированный обратно (см. раздел
**NDP-proxy (ndppd)** ниже).

### Sticky
`ipv6_sticky = true` (с `ipv6_sticky_ttl_secs`, по умолчанию 1800) пиннит один
source на destination IP на время TTL — так, например, Cloudflare-челлендж
`cf_clearance` остаётся валиден между последовательными запросами клиента.
Ротация сохраняется *между* разными destination и по истечении TTL. Без
prefix/interface-source — безвредный no-op.

## Клиент, direct-маршрут (`outline-ws-rust`)

Upstream-трафик клиента (uplink) туннелируется на ваш сервер и source никогда
не ротируется. Эти настройки касаются только **direct-маршрута** (`via =
"direct"` / обход TUN), где клиент сам ходит в интернет.

### `prefer_public_ipv6_src` — стабильный source *(по умолчанию `true`)*
Запрашивает `IPV6_PREFER_SRC_PUBLIC` (RFC 5014) на исходящих IPv6-сокетах,
чтобы ядро выбрало **стабильный public/SLAAC-адрес** вместо ротируемого
temporary privacy-extensions. Это не даёт direct-IPv6-соединениям рваться,
когда у temporary-адреса истекает `valid_lft` и ядро его удаляет.
**Поставьте `false`**, чтобы вернуться к ротации privacy-extensions как
source — т.е. намеренно ротировать, принимая, что долгие flow рвутся при
истечении `valid_lft`. Системный `use_tempaddr` больше на это не влияет —
единственный источник истины это ключ конфига. Linux only, best-effort.

### `direct_ipv6_prefix_interface` — ротируемый /64 *(opt-in)*
Клиентское зеркало серверного `outbound_ipv6_prefix_interface`: каждый
direct-IPv6-дайл биндит случайный адрес из текущего /64 интерфейса (читается
per-connect из `/proc/net/if_inet6`, поэтому следует за динамическим
префиксом). То же требование к обратному трафику — /64 должен быть
маршрутизирован обратно через ndppd. Если задан, для direct-дайлов имеет
приоритет над `prefer_public_ipv6_src`.

## Как выбрать режим

```
Контролируете ли вы префикс, МАРШРУТИЗИРОВАННЫЙ на хост (DHCPv6-PD, AnyIP)?
├── Да, и он статический               → outbound_ipv6_prefix
├── Да, но он меняется (PD/реконнект)   → *_prefix_interface  (+ ndppd)
└── Нет — обычный общий SLAAC-сегмент
        ├── нужна ротация по всему /64  → *_prefix_interface + ndppd
        │     (только если вышестоящий/роутер НЕ делает строгий RPF — сначала тест)
        ├── нужна ротация, NDP/RPF не решить → outbound_ipv6_interface
        │     (ротация только по реальным temporary-адресам)
        └── нужна стабильность (без ротации) → клиент: prefer_public_ipv6_src
```

Типичная реальная топология: провайдер → Keenetic (получает префикс по
DHCPv6-PD, меняющийся при реконнекте) → SLAAC /64 в LAN → хост. Хост видит
динамический `proto ra` /64 на общем сегменте, поэтому режимы «весь /64»
требуют ndppd, а префикс надо re-derive (`*_prefix_interface` именно это и
делают).

## NDP-proxy (ndppd) для prefix-режимов

Prefix-режимы (`outbound_ipv6_prefix`, `*_prefix_interface` и клиентский
`direct_ipv6_prefix_interface`) биндят случайные, **не сконфигурированные**
адреса. На SLAAC-сегменте хост должен отвечать NDP за весь /64, чтобы ответы
возвращались. Это делает `ndppd`. Поскольку /64 динамический,
проксируемый префикс надо держать в синхронизации —
`scripts/ndppd-prefix-sync.{sh,service,timer}` re-derive текущий RA `/64` и
переписывает `ndppd.conf` при смене:

```sh
apt install ndppd
install -m 0755 scripts/ndppd-prefix-sync.sh /usr/local/sbin/
install -m 0644 scripts/ndppd-prefix-sync.service scripts/ndppd-prefix-sync.timer /etc/systemd/system/
# ndppd нужны forwarding + proxy_ndp (закрепите в /etc/sysctl.d/):
sysctl -w net.ipv6.conf.all.forwarding=1
sysctl -w net.ipv6.conf.enp1s0.proxy_ndp=1
systemctl daemon-reload
systemctl enable --now ndppd ndppd-prefix-sync.timer
```

## Ротация vs долгие соединения

Это настоящий, неустранимый компромисс при коротком времени жизни префикса:

- Режим **пула интерфейса** ротирует по temporary-адресам ядра. Долгое
  соединение живёт лишь пока у его source-адреса не истечёт `valid_lft` и ядро
  его не удалит — тогда оно рвётся. Кодом это не предотвратить; управляется RA
  `valid_lft`, который задаёт вышестоящий узел.
- Если можете поднять RA `valid_lft` (свой роутер): задайте `temp_prefered_lft`
  коротким (частая ротация для новых соединений) и `temp_valid_lft` длинным
  (старые адреса задерживаются, существующие flow выживают). Потолок
  `temp_valid_lft` — RA-анонсированный `valid_lft`.
- Если время жизни префикса короткое и неуправляемое, ротация и стабильность
  долгих flow физически конфликтуют — выбирайте на хост настройками выше.
- Режимы `*_prefix_interface` всегда берут как префикс **preferred** (никогда
  не deprecated) адрес, что минимизирует, как часто свежий flow попадёт на
  адрес, который вот-вот исчезнет.

## Проверка

Прежде чем полагаться на любой prefix-режим «весь /64», убедитесь, что
вышестоящий узел принимает не-сконфигурированный source (NDP + RPF):

```sh
ip -6 addr add 2a01:…:dead/64 dev enp1s0 nodad
ping6 -c3 -I 2a01:…:dead 2606:4700:4700::1111   # есть ответы → NDP/RPF OK
ip -6 addr del 2a01:…:dead/64 dev enp1s0
```
Нет ответа → сегмент не вернёт трафик для произвольных source; используйте
`outbound_ipv6_interface` (сервер) или `prefer_public_ipv6_src` (клиент).

Полезное состояние:
```sh
ip -6 route show proto ra                 # текущий динамический /64 (и его expiry)
ip -6 addr show dev enp1s0                 # temporary vs mngtmpaddr, lifetimes
sysctl net.ipv6.conf.{all,default}.use_tempaddr   # включена ли ротация?
```

## Справочник конфига

Сервер (`outline-ss-rust`, блок `[outbound]` или флаги `--outbound-ipv6-*`):

| Ключ | По умолчанию | Значение |
|------|--------------|----------|
| `ipv6_prefix` | не задан | статический CIDR-пул source |
| `ipv6_interface` | не задан | пул адресов интерфейса |
| `ipv6_prefix_interface` | не задан | динамический /64 из интерфейса |
| `ipv6_refresh_secs` | 30 | интервал re-enumerate / re-derive |
| `ipv6_sticky` | true | пин source на destination |
| `ipv6_sticky_ttl_secs` | 1800 | TTL sticky |

Клиент (`outline-ws-rust`, верхний уровень конфига):

| Ключ | По умолчанию | Значение |
|------|--------------|----------|
| `prefer_public_ipv6_src` | `true` | стабильный public source для direct-IPv6; `false` → ротация privacy-extensions |
| `direct_ipv6_prefix_interface` | не задан | ротируемый /64 source для direct-IPv6 (нужен ndppd) |
| `direct_fwmark` | не задан | SO_MARK для direct-сокетов (анти-loopback) |
