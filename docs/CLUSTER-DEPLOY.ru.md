# Runbook развёртывания кластера

Операционное руководство: как превратить два и более отдельных сервера
`outline-ss-rust` в mesh-кластер, где клиентская сессия выживает при смене
edge. Дизайн — в [`CLUSTER.ru.md`](CLUSTER.ru.md).

## 0. Предпосылки

- **Две и более ноды.** Кластер из одной ноды бессмыслен — некому релеить.
  Обычно ноды в разных странах.
- **Все ноды одной сборки.** У mesh-протокола есть версия (`OPEN_VERSION`);
  разнобой сборок ломает mesh-handshake или разбор OPEN. Раскатывай один бинарь
  на всё.
- Mesh-интерконнект — **QUIC поверх UDP**. Убедись, что UDP между нодами
  разрешён (см. §4).

## 1. Топология и план shard'ов

Каждая нода одновременно **edge** (принимает клиентов) и **home** (владеет
своими сессиями и принимает релеи с других edge). Присвой каждой уникальный
`shard_id` в диапазоне `0..16`:

| Нода | shard_id | публичный вход (WS/H3/XHTTP) | mesh-адрес (нода-нода) |
| --- | --- | --- | --- |
| A (напр. DE) | 0 | без изменений | `A_ip:9443` |
| B (напр. NL) | 1 | без изменений | `B_ip:9443` |

## 2. Общий секрет

Один секрет на весь кластер, одинаковый на каждой ноде:

```bash
openssl rand -base64 32
```

Он HKDF-разделяется по доменам (`shard-obfuscation` для session id, `mesh-auth`
для keypair интерконнекта) — **ни CA, ни сертификатов** раздавать не нужно.
Храни как любой секрет (не в git). Утечка ломает будущий mesh-auth, но прошлый
трафик защищён эфемерными QUIC-ключами.

## 3. Серверный конфиг `[cluster]`

`peers` — полная карта shard → mesh-адрес **всех** нод; собственная запись ноды
игнорируется, поэтому один и тот же блок `peers` можно копировать на все ноды.

**Нода A** (`shard_id = 0`):

```toml
[cluster]
enabled = true
shard_id = 0
cluster_psk = "<тот же base64 на каждой ноде>"
mesh_listen = "[::]:9443"          # QUIC (UDP) слушатель входящих релеев
mesh_relay_budget_ms = 4000        # релей, стоящий дольше, рвём
peers = [
  { shard = 0, addr = "A_ip:9443" },   # своя — игнорируется
  { shard = 1, addr = "B_ip:9443" },
]
```

**Нода B** — то же самое, но `shard_id = 1`.

Валидация fail-fast при старте: `shard_id` обязателен и `< 16`; `cluster_psk` —
валидный непустой base64; `mesh_listen` — валидный `host:port`; дубликат shard в
`peers` = ошибка. `enabled = false` (или отсутствие всей секции) = standalone,
байт-в-байт текущее поведение.

## 4. Сеть / firewall

- Открой **mesh-порт (9443/UDP)** между нодами. Это QUIC — **UDP**, не TCP.
- Defense in depth: ограничь mesh-порт IP пиров. PSK-derived mutual pin и так
  отсекает чужих, но IP-фильтр сужает поверхность.
- Публичный вход (WS/H3/XHTTP) — без изменений.

## 5. Клиент: как достучаться до кластера

Клиент **кластер-агностичен и не требует правок кода**, но способ адресации
кластера важен. Клиент кэширует resume id по **resume-scope**; сессия выживает
при смене edge только если клиент предъявляет *один и тот же* resume id тому
edge, куда попал. Есть три способа это обеспечить, по убыванию предпочтения:

### 5a. Anycast (идеал)

Один IP, анонсируемый по BGP со всех нод. Клиент всегда дайлит один адрес; сеть
направляет на ближайшую/живую ноду. Один scope, один resume id, выживание
даром. В клиентском конфиге ничего особого.

### 5b. Один DNS-hostname

Один hostname с несколькими A/AAAA-записями. Работает, **если** клиент
сконфигурен одним uplink-URL на этот hostname (scope = uplink, не resolved IP).
Ненадёжно при нестабильном резолвинге.

### 5c. Явные uplink'и с `shared_resume` (нет anycast / нет надёжного DNS)

Пропиши каждую ноду как отдельный uplink и пометь группу так, чтобы все uplink'и
**делили один resume-scope** (имя группы). Тогда, на какой бы edge ни дайлил
клиентский балансировщик, он предъявляет один resume id, edge релеит сессию на
её home, и она выживает при смене.

Простая (неявная группа) форма:

```toml
[[uplinks]]
name = "edge-de"
transport = "ss"                   # или vless; см. UPLINK-CONFIGURATIONS.ru.md
# ...url / cipher / password / mode для ноды A...

[[uplinks]]
name = "edge-nl"
transport = "ss"
# ...то же для ноды B...

[load_balancing]
shared_resume = true               # ← все uplink'и выше делят один resume id
```

Форма с именованными группами (`[[uplink_group]]`) — выстави `shared_resume =
true` на группе, чьи uplink'и являются edge'ами кластера.

`shared_resume` по умолчанию `false`. **Включай только для группы, чьи uplink'и —
edge'ы одного mesh-кластера.** Для группы независимых серверов общий resume id
между несвязанными home'ами всегда давал бы miss.

Общий scope охватывает **и UDP, и TCP**: при `shared_resume = true` сессии
SS-UDP и VLESS-UDP предъявляют групповой shard-несущий resume id и мигрируют
через переключение edge так же, как TCP (релеятся на свой home-shard). См.
заметку «Cross-node миграция UDP» в
[`UPLINK-CONFIGURATIONS.ru.md`](../bins/outline-ws-rust/docs/UPLINK-CONFIGURATIONS.ru.md).

## 6. Порядок раскатки

1. Сгенерь PSK; подготовь блоки `[cluster]`.
2. Открой mesh-порт между нодами.
3. Раскатай новый бинарь + конфиг на все ноды (rolling ок — нода на старом
   бинаре просто работает standalone для своих клиентов; безопасная деградация).
4. Направь клиента на кластер (§5).
5. Прогони проверки ниже.

## 7. Проверка

- **Старт:** в логе каждой ноды mesh listener поднялся, нет паники валидации
  конфига.
- **Mesh-связность:** с тестового клиента подключись к одной ноде, предъявив
  resume id, декодирующийся в shard *другой* ноды (в реале это происходит само
  при переезде клиента между edge); сессия обслуживается, upstream не
  пересоздаётся. При недоступном home edge тихо деградирует к fresh-сессии
  (топология не раскрывается).
- **Выживание:** установи сессию (идёт загрузка), принудительно переведи клиента
  на другую ноду (§5c делает это детерминированным — убей активный uplink), и
  загрузка продолжается.
- **Метрики:** на `/metrics` каждой ноды (и в Grafana-дашборде ss-rust, ряд
  *Cluster Mesh*):
  - рост `outline_ss_mesh_relay_opened_total{outcome="ok"}` ⇒ edge'ы релеят
    cross-shard сессии на свои home; рост `{outcome="fail"}` ⇒ пир недостижим
    (нода упала, mesh UDP-порт закрыт или несовпадение PSK), и edge деградирует
    к свежим локальным сессиям.
  - `outline_ss_mesh_relay_active` — сколько relay home-нода обслуживает прямо
    сейчас (ноль на праздном кластере — норма: mesh-стримы открываются по
    требованию, не держатся). Плюс смотри `outline_ss_orphan_resume_hit_total`
    на home: растёт каждый раз, когда home переприкрепляет припаркованный
    upstream для клиента, пришедшего через другой edge.
  - рост `outline_ss_mesh_relay_rejected_total{reason="capacity"}` ⇒ home упёрся
    в лимит одновременных релеймых сессий и отвергает новые relay-стримы (edge'ы
    деградируют к свежим локальным сессиям). Норма — ноль; устойчивый рост
    означает, что кластер шлёт на одну home больше одновременных релеймых
    сессий, чем она рассчитана держать.
  - **Кластерный трафик** (сколько данных реально идёт через mesh, а не только
    сколько relay открыто): `outline_ss_mesh_bytes_total{role,direction,transport}`
    и `outline_ss_mesh_datagrams_total{role,direction}`. `role="edge"` — трафик,
    который эта нода форвардит в кластер; `role="home"` — который она обслуживает
    для чужих edge'ей: одна и та же relayed-сессия, посчитанная с двух концов.
    Ноль на обоих = трафик через mesh не идёт (все сессии локальны). Панели
    *Mesh Throughput — edge/home* и *Mesh Datagram Rate*.
  - `outline_ss_mesh_throttle_hints_sent_total` /
    `outline_ss_mesh_throttle_hints_received_total{outcome}` /
    `outline_ss_mesh_control_datagram_errors_total` — throttle-сигналинг edge→home;
    устойчивый `received{outcome="dropped"}` или `control_datagram_errors` ⇒
    расхождение конфига/версий между edge и home. Панель *Mesh Throttle Hints &
    Control Errors*.
  - На **клиенте** (дашборд ws-rust, ряд *Cluster / Soft-switch*):
    `outline_ws_soft_switch_total{outcome}` — операторские soft-switch
    миграции, на здоровом переключении доминирует `migrated`; и
    `outline_ws_resume_lookup_total{transport,scope,result}` —
    `scope="group",result="hit"` = cross-node-способный resume, предъявленный
    новому edge.
- **⚠️ Целостность на боевом трафике:** e2e-тесты покрывают data-plane, но не
  боевой трафик. Скачай большой файл через кластер и **сверь sha256** с
  оригиналом (риск тихой порчи / переупорядочивания, как с TUN pump). Держи
  `git revert` наготове первые дни.

## 8. Откат

- Мгновенный: `enabled = false` в `[cluster]` (или удалить секцию) + рестарт —
  нода снова standalone, session id снова plain-random, mesh не слушает. Клиенты
  продолжают работать (деградируют к fresh-сессии). На клиенте убери
  `shared_resume` (или направь на одну ноду).
- Полный: revert кластерных коммитов и пересборка. Но для выключения достаточно
  `enabled = false` — код без `[cluster]` неизменён.

## 9. Оговорки

- **Double-hop RTT:** хоп edge → home между странами добавляет задержку на
  длинных bulk-передачах. Health-budget ловит *зависания*, не медленность.
- **Throttle-detection:** держи **выключенным** на старте — детектор на home не
  отличает mesh-интерконнект от client-last-mile и может ложно срабатывать (см.
  `CLUSTER.ru.md`).
- **UDP-носители** пока не релеятся — эти ноги деградируют к
  fresh-сессии на чужом shard (безопасно, просто нет cross-edge resume).
