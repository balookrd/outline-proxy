# Carrier-padding

*English version: [PADDING.md](PADDING.md)*

Адаптивный application-layer padding для носителей WebSocket / XHTTP. Ломает
корреляцию по размеру TLS-записей, на которую опираются классификаторы
«proxy-inside-TLS» (TLS-in-TLS), не внедряя второй прокси-протокол — та же цель,
что у padding AnyTLS, но встроенная в уже имеющиеся в `outline-proxy` носители.

## Зачем

Когда проксируемый поток туннелируется внутри TLS (Shadowsocks или VLESS поверх
WebSocket / XHTTP), каждая запись приложения, как правило, ложится в одну внешнюю
TLS-запись. Тогда *размер* этой записи повторяет размер внутреннего
проксируемого чанка. Пассивный классификатор, который не может читать
TLS-открытый текст, всё равно видит последовательность размеров записей и узнаёт
характерную форму проксируемого потока, вложенного в TLS, — эвристика,
применяемая, среди прочего, российским ТСПУ. У VLESS-over-WS / XHTTP ровно та же
уязвимость, что у Shadowsocks, поэтому он паддится так же.

Padding развязывает эти две величины: каждый проксируемый чанк (зашифрованный
Shadowsocks-чанк или VLESS-кадр) оборачивается в кадр с префиксом длины, чей
хвост — случайное число выбрасываемых байт, так что буфер, передаваемый
TLS-слою, больше не имеет фиксированной связи с переносимой полезной нагрузкой.

## Формат кадра

Каждый кадр — `real_len | pad_len | real | pad`:

| Поле       | Размер           | Значение                                |
|------------|------------------|-----------------------------------------|
| `real_len` | `u16` big-endian | длина реальных проксируемых байт        |
| `pad_len`  | `u16` big-endian | длина pad-хвоста                        |
| `real`     | `real_len` байт  | зашифрованный SS-чанк или VLESS-кадр     |
| `pad`      | `pad_len` байт   | случайные байты, отбрасываются на декоде |

Кодек живёт в `crates/outline-wire/src/padding.rs` и является чистым
фреймингом — без RNG и без часов; источник случайности (pad-байты и случайный
розыгрыш их длины) предоставляет вызывающий транспортный слой, зеркаля кодек
SS2022-заголовка. Декодер (`PaddingDecoder`) — потоковый автомат: вход может
быть разрезан на любой границе байт, потому что h2 / h3 DATA-кадры не связаны с
границами наших кадров. Кадр с `real_len = 0` несёт только pad — это форма
**cover-кадра** (см. ниже); декодер ничего из него не выдаёт.

`real` и `pad` ограничены 65535 байтами (потолок `u16`); транспорт чанкует более
крупную запись до этой границы перед фреймингом, и pad несёт только последний
чанк записи — так что одна транспортная запись даёт один случайный хвост,
независимо от числа сегментов.

## Гейт: config-синхронизация, без согласования

**On-wire-бита нет.** Пир, не включивший padding, просто никогда не фреймит свои
записи и не декодирует; пир, который включил, скормит обычный (нефреймированный)
поток в декодер и упадёт. Поэтому **обе стороны должны включить опцию вместе** —
ровно как cross-transport session resumption. По умолчанию выключено, что
оставляет wire байт-в-байт совпадающим с непаддингованным носителем, поэтому
сторонние клиенты (Happ, Outline, xray, sing-box) не затронуты, пока оператор не
включит padding.

- **Сервер — per-path.** `[padding] paths` перечисляет carrier-пути, которые
  паддятся. Фреймятся только соединения, чей путь попадает в этот набор; все
  остальные пути сохраняют чистый Shadowsocks / VLESS wire. Это позволяет одному
  серверу паддить своих клиентов на одном пути (SS *или* VLESS), обслуживая
  сторонних клиентов (включая сторонние VLESS-клиенты вроде xray / sing-box) на
  другом без изменений.
- **Клиент — глобальный дефолт + per-uplink override.** Глобальный блок
  `[padding]` задаёт параметры схемы (диапазон / cover / jitter) и дефолтный
  on/off — `enabled`. Каждый `[[outline.uplinks]]` может переопределить on/off
  через `padding = true` / `padding = false`: эффективное решение для dial — это
  per-uplink значение, если задано, иначе глобальный дефолт `enabled` (та же
  схема override/fallback, что у per-uplink `fingerprint_profile`). Так оператор
  может оставить глобальный дефолт выключенным и паддить только аплинки, ведущие
  на его собственные серверы (`padding = true`), либо оставить включённым и
  исключить конкретный аплинк — например VLESS-аплинк к стороннему серверу
  xray / sing-box — через `padding = false`. Padded-dial (и SS, и VLESS, и TCP, и
  UDP) обязан указывать на серверные пути, которые тоже паддятся.

## Cover-трафик

При `cover = true` простаивающее соединение шлёт pad-only **cover-кадры**
(`real_len = 0`) со случайным интервалом, равномерно выбранным из
`[cover_jitter_min_ms, cover_jitter_max_ms]`, чтобы тишина не выдавала тайминг.
Таймер сбрасывается после каждой реальной записи, поэтому cover-кадры
срабатывают только на действительно простаивающем канале и никогда не
перемежаются с живым трафиком. Клиент шлёт cover на uplink-writer'е; сервер — на
downlink-writer'е. Cover-кадр — это бинарный data-кадр (никогда не WebSocket
Ping), поэтому он безопасен на H3-носителе: он не может эскалировать до
connection-level `H3_INTERNAL_ERROR`, которым рискует server-originated Ping.
Декодер получателя прозрачно отбрасывает cover-кадры.

## Покрытые носители

Padding едет на уровне `Message`, общем для всех носителей WS / XHTTP, поэтому
один механизм покрывает их все:

- **SS-over-WebSocket** — h1, h2, **h3** (WebSocket-over-HTTP/3, RFC 9220).
- **SS-over-XHTTP** — h1, h2, h3.
- **SS-UDP поверх WebSocket / XHTTP** — паддится **по датаграмме** (один кадр на
  один зашифрованный Shadowsocks-пакет); см. *UDP* ниже.
- **VLESS-over-WebSocket** — h1, h2, **h3**.
- **VLESS-over-XHTTP** — h1, h2, h3.
- **VLESS-UDP поверх WebSocket** — паддится **по датаграмме** (один кадр на один
  пакет); см. *UDP* ниже.

### UDP

Все UDP-носители паддятся одинаково, так что асимметрии «SS-UDP / VLESS-UDP»
больше нет. Каждая датаграмма оборачивается ровно в один кадр (кодек никогда не
разрезает пакет на несколько датаграмм, поэтому границы пакетов сохраняются), а
получатель прогоняет каждую входящую датаграмму через потоковый декодер до
парсинга SS / VLESS-слоем. Cover-кадр (`real_len = 0`) на тихом downlink
декодируется в пусто и отбрасывается. Raw-сокет и raw-QUIC datagram-канал — это
*не* WS-носители и остаются чистыми (см. *Вне scope*).

- **VLESS-UDP** мультиплексирует TCP и UDP на *одном* пути (различая их по
  command-байту *внутри* первого кадра), поэтому сервер не может отличить ноги
  до чтения данных — значит, padded VLESS-путь *обязан* паддить и UDP-ногу.
- **Split SS** разводит TCP и UDP по *разным* путям. Внесите оба в
  `[padding] paths`, чтобы западдить весь аплинк. Клиентский per-uplink
  переключатель работает по принципу «всё или ничего» — padded-аплинк фреймит
  каждую отправляемую датаграмму, — поэтому padded SS-аплинк ожидает, что оба
  его серверных пути (TCP и UDP) тоже паддятся.
- **Combined SS** кладёт TCP и UDP на один base-path, разделяя их по скрытому
  биту в токене (WS) / session id (XHTTP), который сервер декодирует на
  upgrade-этапе. Обе ноги резолвят один base-path, поэтому внесение combined
  base-path в `[padding] paths` паддит *обе* ноги: UDP-нога `run_udp_relay`
  резолвит ту же per-path-схему, что и TCP-нога `run_tcp_relay`.

**Вне scope:**

- **Raw SS / VLESS поверх QUIC** (ALPN `ss` / `vless`) — отдельный транспорт, не
  WS-носитель; у QUIC своя поверхность фингерпринта (отдельный будущий трек).
  Raw-QUIC datagram-канал доходит до того же `UdpWsTransport`, что и WS-носитель,
  но строится с выключенным padding, поэтому остаётся чистым.

## Конфигурация

### Сервер (`outline-ss-rust`)

```toml
[padding]
enabled = true
# WS/XHTTP carrier-пути для паддинга. SS-TCP, SS-UDP, combined SS base-path и
# VLESS едут на одном per-path-переключателе; внесите и SS-UDP-путь, чтобы
# западдить UDP-ногу единообразно.
paths = ["/SECRET/tcp", "/SECRET/udp", "/SECRET/vless"]
min_bytes = 0                            # мин. pad на кадр
max_bytes = 256                          # макс. pad на кадр (0 = без фрейминга)
cover = false                            # idle pad-only cover-кадры (downlink)
cover_jitter_min_ms = 250                # нижняя граница idle-паузы перед cover
cover_jitter_max_ms = 1500               # верхняя граница
```

Валидация отвергает `enabled = true` с пустым `paths`.

### Клиент (`outline-ws-rust`)

```toml
[padding]
enabled = false              # глобальный дефолт (per-uplink `padding` его переопределяет)
min_bytes = 0
max_bytes = 256              # макс. pad на кадр (0 = без фрейминга)
cover = false                # idle pad-only cover-кадры (uplink)
cover_jitter_min_ms = 250
cover_jitter_max_ms = 1500

# Паддить только свой сервер, даже при выключенном глобальном дефолте:
[[outline.uplinks]]
name = "mine"
# … vless_ws_url / tcp_ws_url / и т.п. …
padding = true               # override: паддить дайлы этого аплинка

# Оставить сторонний VLESS-аплинк чистым, даже при включённом глобальном дефолте:
[[outline.uplinks]]
name = "thirdparty"
# … vless_ws_url на сервер xray / sing-box …
padding = false              # override: никогда не паддить этот аплинк
```

Блок `[padding]` задаёт параметры схемы и глобальный дефолт (`enabled`); ключ
`padding` каждого аплинка переопределяет on/off для этого аплинка (отсутствует →
наследует глобальный дефолт). Профиль по умолчанию (`0..256`) лёгкий — ломает
точную корреляцию размеров при минимальном overhead. Поднимите `max_bytes` для
более широкого распределения размеров ценой большего трафика. Включайте `cover`
на обеих сторонах вместе, когда важен анализ таймингов простоя.

## Эксплуатационные заметки

- Раскатывайте гейт на обе стороны одним изменением. Полу-раскатанная пара (одна
  сторона паддит, другая нет) рвёт затронутые сессии, пока обе не совпадут.
- Padding добавляет 4-байтовый заголовок плюс pad-хвост на запись; на лёгком
  профиле это малый ограниченный overhead. Он не меняет согласованный транспорт,
  ALPN или TLS-рукопожатие.
- Padding резолвится из стартового конфига. Изменение `[padding]` требует
  перезапуска затронутого бинаря (hot-reload не поддерживается).
