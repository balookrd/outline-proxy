# Carrier-padding

*English version: [PADDING.md](PADDING.md)*

Адаптивный application-layer padding для носителей WebSocket / XHTTP. Ломает
корреляцию по размеру TLS-записей, на которую опираются классификаторы
«proxy-inside-TLS» (TLS-in-TLS), не внедряя второй прокси-протокол — та же цель,
что у padding AnyTLS, но встроенная в уже имеющиеся в `outline-proxy` носители.

## Зачем

Когда Shadowsocks-поток туннелируется внутри TLS (SS-over-WebSocket,
SS-over-XHTTP), каждая запись приложения, как правило, ложится в одну внешнюю
TLS-запись. Тогда *размер* этой записи повторяет размер внутреннего
Shadowsocks-чанка. Пассивный классификатор, который не может читать TLS-открытый
текст, всё равно видит последовательность размеров записей и узнаёт характерную
форму проксируемого потока, вложенного в TLS, — эвристика, применяемая, среди
прочего, российским ТСПУ.

Padding развязывает эти две величины: каждый Shadowsocks-чанк оборачивается в
кадр с префиксом длины, чей хвост — случайное число выбрасываемых байт, так что
буфер, передаваемый TLS-слою, больше не имеет фиксированной связи с переносимой
полезной нагрузкой.

## Формат кадра

Каждый кадр — `real_len | pad_len | real | pad`:

| Поле       | Размер           | Значение                                |
|------------|------------------|-----------------------------------------|
| `real_len` | `u16` big-endian | длина реальных Shadowsocks-байт         |
| `pad_len`  | `u16` big-endian | длина pad-хвоста                        |
| `real`     | `real_len` байт  | зашифрованный Shadowsocks-чанк          |
| `pad`      | `pad_len` байт   | случайные байты, отбрасываются на декоде |

Кодек живёт в `crates/outline-wire/src/padding.rs` и является чистым
фреймингом — без RNG и без часов; источник случайности (pad-байты и случайный
розыгрыш их длины) предоставляет вызывающий транспортный слой, зеркаля кодек
SS2022-заголовка. Декодер (`PaddingDecoder`) — потоковый автомат: вход может
быть разрезан на любой границе байт, потому что h2 / h3 DATA-кадры не связаны с
границами наших кадров. Кадр с `real_len = 0` несёт только pad — это форма
**cover-кадра** (см. ниже); декодер ничего из него не выдаёт.

`real` и `pad` ограничены 65535 байтами (потолок `u16`); транспорт чанкует более
крупную запись до этой границы перед фреймингом, и pad несёт только последний
чанк записи — так что одна транспортная запись даёт один случайный хвост,
независимо от числа сегментов.

## Гейт: config-синхронизация, без согласования

**On-wire-бита нет.** Пир, не включивший padding, просто никогда не фреймит свои
записи и не декодирует; пир, который включил, скормит обычный (нефреймированный)
поток в декодер и упадёт. Поэтому **обе стороны должны включить опцию вместе** —
ровно как cross-transport session resumption. По умолчанию выключено, что
оставляет wire байт-в-байт совпадающим с непаддингованным носителем, поэтому
сторонние клиенты (Happ, Outline, xray, sing-box) не затронуты, пока оператор не
включит padding.

- **Сервер — per-path.** `[padding] paths` перечисляет carrier-пути, которые
  паддятся. Фреймятся только соединения, чей путь попадает в этот набор; все
  остальные пути сохраняют чистый Shadowsocks-over-WS / XHTTP wire. Это позволяет
  одному серверу паддить своих клиентов на одном пути, обслуживая сторонних
  клиентов на другом без изменений.
- **Клиент — глобально.** Клиент всегда «наш», поэтому тумблер — единое
  process-wide значение: при включении паддится каждый WS / XHTTP dial. Он должен
  указывать на серверный путь, который тоже паддится.

## Cover-трафик

При `cover = true` простаивающее соединение шлёт pad-only **cover-кадры**
(`real_len = 0`) со случайным интервалом, равномерно выбранным из
`[cover_jitter_min_ms, cover_jitter_max_ms]`, чтобы тишина не выдавала тайминг.
Таймер сбрасывается после каждой реальной записи, поэтому cover-кадры
срабатывают только на действительно простаивающем канале и никогда не
перемежаются с живым трафиком. Клиент шлёт cover на uplink-writer'е; сервер — на
downlink-writer'е. Cover-кадр — это бинарный data-кадр (никогда не WebSocket
Ping), поэтому он безопасен на H3-носителе: он не может эскалировать до
connection-level `H3_INTERNAL_ERROR`, которым рискует server-originated Ping.
Декодер получателя прозрачно отбрасывает cover-кадры.

## Покрытые носители

Padding едет на уровне `Message`, общем для всех носителей WS / XHTTP, поэтому
один механизм покрывает их все:

- **SS-over-WebSocket** — h1, h2, **h3** (WebSocket-over-HTTP/3, RFC 9220).
- **SS-over-XHTTP** — h1, h2, h3.

**Вне scope:**

- **UDP-носители** (SS-UDP поверх WS / XHTTP) — один Shadowsocks-пакет на кадр;
  padding — функция TCP-потока и не применяется.
- **Raw SS / VLESS поверх QUIC** (ALPN `ss` / `vless`) — отдельный транспорт, не
  WS-носитель; у QUIC своя поверхность фингерпринта (отдельный будущий трек).
- **VLESS**-носители не паддятся.

## Конфигурация

### Сервер (`outline-ss-rust`)

```toml
[padding]
enabled = true
paths = ["/SECRET/tcp", "/SECRET/ss"]   # WS/XHTTP carrier-пути для паддинга
min_bytes = 0                            # мин. pad на кадр
max_bytes = 256                          # макс. pad на кадр (0 = без фрейминга)
cover = false                            # idle pad-only cover-кадры (downlink)
cover_jitter_min_ms = 250                # нижняя граница idle-паузы перед cover
cover_jitter_max_ms = 1500               # верхняя граница
```

Валидация отвергает `enabled = true` с пустым `paths`.

### Клиент (`outline-ws-rust`)

```toml
[padding]
enabled = true
min_bytes = 0
max_bytes = 256              # макс. pad на кадр (0 = без фрейминга)
cover = false                # idle pad-only cover-кадры (uplink)
cover_jitter_min_ms = 250
cover_jitter_max_ms = 1500
```

Профиль по умолчанию (`0..256`) лёгкий — ломает точную корреляцию размеров при
минимальном overhead. Поднимите `max_bytes` для более широкого распределения
размеров ценой большего трафика. Включайте `cover` на обеих сторонах вместе,
когда важен анализ таймингов простоя.

## Эксплуатационные заметки

- Раскатывайте гейт на обе стороны одним изменением. Полу-раскатанная пара (одна
  сторона паддит, другая нет) рвёт затронутые сессии, пока обе не совпадут.
- Padding добавляет 4-байтовый заголовок плюс pad-хвост на запись; на лёгком
  профиле это малый ограниченный overhead. Он не меняет согласованный транспорт,
  ALPN или TLS-рукопожатие.
- Padding резолвится из стартового конфига. Изменение `[padding]` требует
  перезапуска затронутого бинаря (hot-reload не поддерживается).
