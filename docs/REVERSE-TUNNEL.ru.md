# Обратный туннель (топология A)

Этот документ описывает функцию обратного туннеля, которая позволяет серверу
`outline-ss-rust` работать **за NAT без публичного IP**, самостоятельно
дозваниваясь до публичного клиента `outline-ws-rust`, который его слушает.

*English version: [REVERSE-TUNNEL.md](REVERSE-TUNNEL.md)*

## Зачем

В обычном («прямом») развёртывании сервер публичен, а клиент дозванивается до
него. Иногда всё наоборот: машина с «чистым» egress-IP (тем, с которого должен
выходить трафик) стоит за NAT, а единственный публичный хост, которым вы
управляете, — это тот, к которому подключаются пользователи. Топология A это
решает:

- `outline-ss-rust` (**сервер** за NAT) дозванивается *наружу* по QUIC к
  публичному `outline-ws-rust`.
- `outline-ws-rust` (**клиент**, публичный VPS) **слушает**, принимает несущую
  и маршрутизирует пользовательский SOCKS5/TUN-трафик через неё.
- Трафик пользователей входит на `ws`; в интернет он выходит с egress-IP `ss`.

## Как это работает

Инвертируется только **направление несущей**. Stream-уровневый data plane не
меняется относительно прямого пути, потому что QUIC позволяет любому пиру
открывать двунаправленные стримы независимо от того, кто дозвонился:

| Роль | Прямое развёртывание | Обратное (топология A) |
|------|----------------------|------------------------|
| `ss` | QUIC-сервер (`accept`) | QUIC-**клиент** (`connect`), по-прежнему `accept_bi` |
| `ws` | QUIC-клиент (`connect`) | QUIC-**сервер** (`accept`), открывает `bi` на сессию |
| TLS  | `ss` предъявляет server cert | инвертируется: `ws` предъявляет server cert, `ss` — client cert (mTLS) |

Каждая пользовательская сессия — это один QUIC bidi-стрим, несущий либо
«сырой» Shadowsocks (SS-AEAD / SS-2022), либо VLESS, ровно как прямой
raw-QUIC транспорт — протокол выбирается **per-peer** и согласуется по ALPN
несущей (`ss`/`ss-mtu` либо `vless`/`vless-mtu`), так что один слушатель
несёт и SS-, и VLESS-пиров. UDP едет в QUIC-датаграммах (VLESS мультиплексирует
per-target сессии по id). Сервер `ss` переиспользует существующие accept-циклы
raw-SS / raw-VLESS без изменений — им всё равно, что несущую дозвонили наружу.

### Аутентификация: mTLS + pinned-сертификаты

CDN-фронтинг к обратной несущей неприменим, поэтому ни одна из сторон не
использует публичное хранилище доверия webpki. Вместо этого обе стороны пинят
сертификат контрагента по **SHA-256 fingerprint**, и слушатель требует
клиентский сертификат (взаимный TLS):

- `ss` предъявляет **клиентский сертификат**; `ws` принимает несущую, только
  если fingerprint этого серта в его allow-list.
- `ws` предъявляет **серверный сертификат**; `ss` завершает handshake, только
  если fingerprint этого серта совпадает с настроенным пином.

Fingerprint клиентского серта пира также определяет, *какой* настроенный пир
подключился, а значит — какие креды кадрирования слушатель использует для
стримов этого пира (Shadowsocks `method` / `password` для SS-пира либо UUID
`vless_id` для VLESS-пира) и в какую egress-группу (`group`) он попадает.

## Настройка

### 1. Сгенерировать пару сертификатов

Нужны два self-signed серта: один предъявляет слушатель `ws`, другой —
дозвонщик `ss`. Подойдёт любой инструмент; с OpenSSL:

```bash
# серверный сертификат ws (предъявляет публичный слушатель)
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -nodes \
  -keyout ws-server.key -out ws-server.crt -days 3650 \
  -subj "/CN=reverse" -addext "subjectAltName=DNS:reverse"

# клиентский сертификат ss (предъявляет дозвонщик за NAT)
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -nodes \
  -keyout ss-client.key -out ss-client.crt -days 3650 \
  -subj "/CN=reverse-ss" -addext "subjectAltName=DNS:reverse-ss"
```

Вычислите SHA-256 fingerprint каждого серта над его **DER**-кодировкой — это
строка пина, которую настраивает противоположная сторона (hex, опционально
через двоеточие, или base64 от 32 байт):

```bash
openssl x509 -in ws-server.crt -outform DER | openssl dgst -sha256
openssl x509 -in ss-client.crt -outform DER | openssl dgst -sha256
```

### 2. Настроить публичный слушатель `ws`

```toml
[reverse_listener]
enabled = true
# UDP-адрес, на который биндится QUIC server endpoint.
listen = "0.0.0.0:8443"
# Сертификат, который этот слушатель предъявляет дозвонщикам.
server_cert_path = "/etc/outline-ws/ws-server.crt"
server_key_path  = "/etc/outline-ws/ws-server.key"
# Группа по умолчанию, в которую сводятся reverse-пиры (per-peer `group`
# её переопределяет). Направьте трафик сюда через [[route]].
group = "reverse"
# true (по умолчанию) предлагает ss-mtu, затем ss (включает oversize-record fallback).
mtu = true
# Верхняя граница числа одновременно зарегистрированных пиров, на каждую
# группу (по умолчанию 8).
max_peers = 8

# По одной записи на каждый ожидаемый пир. `client_cert_pin` аутентифицирует
# несущую (mTLS). Протокол выбирается per-peer: SS-пир задаёт `method` +
# `password` (должны совпадать с пользователем ss); VLESS-пир задаёт `vless_id`
# (UUID, должен совпадать с пользователем ss `[vless]`) — ровно одна из двух
# форм. `group` необязателен — опустите, чтобы попасть в listener-level `group`,
# или задайте, чтобы развести разных пиров по разным egress-группам.
[[reverse_listener.peers]]
client_cert_pin = "aa:bb:cc:..."          # SHA-256 от ss-client.crt (DER)
method   = "2022-blake3-aes-256-gcm"      # SS-пир
password = "<base64-psk-или-пароль>"
# group опущен → попадает в listener-дефолт ("reverse").

[[reverse_listener.peers]]
client_cert_pin = "11:22:33:..."          # второй пир, VLESS, отдельный egress
vless_id = "b831381d-6324-4d53-ad4f-8cda48b30811"   # должен совпадать с ss [vless]-пользователем
group    = "reverse-eu"                    # пул и маршрут отдельные
```

VLESS-пир выше дозванивается с ALPN `vless`/`vless-mtu`, SS-пир — с
`ss`/`ss-mtu`. Единый слушатель предлагает оба набора и направляет каждую
несущую в accept-цикл её протокола.

Reverse-группа — **первоклассная цель маршрута**: достаточно объявить её на
`[reverse_listener]` (его `group` плюс per-peer `group`), чтобы на неё
маршрутизировать — одноимённый `[[uplink_group]]` НЕ нужен. Просто направьте
`[[route]]`:

```toml
[[route]]
# Конкретные адреса выходят через второй пир.
prefixes = ["203.0.113.0/24"]
via = "reverse-eu"

[[route]]
# Всё остальное выходит через пир по умолчанию.
default = true
via = "reverse"
```

Когда для reverse-группы нет подключённого пира, резолв идёт по порядку:
одноимённый `[[uplink_group]]`, если вы его объявили (явный fallback на обычные
uplink'и, пока пир оффлайн); иначе `fallback_via` / `fallback_direct` /
`fallback_drop` самого маршрута; иначе сессия дропается. Никогда молча не
утекает в дефолтную группу.

> **Примечание:** `ws` должен объявить хотя бы одну обычную uplink-группу
> (`[[uplink_group]]` + `[[outline.uplinks]]`), даже если ваш единственный
> egress — это reverse-пиры: на одном `[reverse_listener]` прокси не
> стартует. Достаточно простой forward-группы (её uplink'и не используются,
> пока трафик идёт в reverse-группу); в будущем требование могут смягчить.

### 3. Настроить дозвонщик `ss` (за NAT)

```toml
[reverse_tunnel]
enabled = true

[[reverse_tunnel.endpoints]]
# host:port публичного слушателя ws. host — DNS-имя или литеральный IP.
addr = "ws.example.com:8443"
# TLS SNI / server name. По умолчанию — host-часть `addr`.
server_name = "reverse"
# SHA-256 от ws-server.crt (DER) — пинит серт слушателя (без webpki).
server_cert_pin = "dd:ee:ff:..."
# Клиентский сертификат для mTLS (его пин в allow-list на ws).
client_cert_path = "/etc/outline-ss/ss-client.crt"
client_key_path  = "/etc/outline-ss/ss-client.key"
# Протокол несущей: "ss" (по умолчанию) или "vless". Должен совпадать с
# протоколом соответствующей записи пира на слушателе ws; выбирает ALPN
# (ss/ss-mtu либо vless/vless-mtu) и accept-цикл.
protocol = "ss"
# true (по умолчанию) предлагает -mtu ALPN-сиблинг первым; false — только базовый.
mtu = true
# Пол / потолок backoff'а переподключения в секундах для транзиентных
# сбоев (по умолчанию 1 / 60).
backoff_min_secs = 1
backoff_max_secs = 60
```

Пользователь, кадрирующий трафик (`method` / `password` либо `vless_id` в
записи пира на `ws`), должен существовать в `[[users]]` сервера `ss` (для SS)
или среди `[vless]`-пользователей (для VLESS), чтобы прошёл handshake —
mTLS-identity и SS/VLESS-пользователь это независимые слои.

Каждый endpoint крутит свой цикл переподключения с ограниченным
джиттер-backoff'ом. Некорректный пин или нечитаемый сертификат отключает
только этот endpoint (с логом при старте), не роняя сервер.

Цикл различает **транзиентные сбои** (хост `ws` лежит, таймаут, сетевой
сбой) и **auth-провалы** (mismatch пина/сертификата — неважно, `ss` ли
отверг серверный серт `ws`, или `ws` отверг клиентский серт `ss`).
Транзиентные сбои ретраятся по экспоненте `backoff_min_secs`→
`backoff_max_secs`; auth-провал повтором не чинится, поэтому цикл уходит в
длинный фиксированный интервал (5 минут) и пишет warning «проверьте
пины/серты», вместо того чтобы долбить пира TLS-хендшейками. Он продолжает
перепроверять с этим интервалом, так что после исправления проблемы с
сертификатом (на `ss` это означает рестарт — reverse-конфиг не
hot-reload'ится) несущая восстанавливается сама.

## Наблюдаемость

| Метрика | Сторона | Смысл |
|---------|---------|-------|
| `outline_ss_reverse_tunnel_active_connections` | ss | Gauge установленных несущих. |
| `outline_ss_reverse_tunnel_connects_total{result}` | ss | Исходы дозвона (`success` / `failure`). |
| `outline_ws_rust_reverse_peers{group}` | ws | Gauge подключённых пиров по группам. |

Лейблы низко-кардинальны; fingerprint'ы сертификатов пиров никогда не
логируются и не экспортируются.

## Ограничения

- **«Сырой» Shadowsocks и VLESS** несутся на обратной несущей (и TCP, и UDP).
  HTTP/3-WebSocket в обратном режиме не несётся.
- **Без CDN-фронтинга** — несущая это прямое QUIC-соединение, аутентифицируемое
  pinned-сертификатами, а не frontable HTTPS-запрос.
- Пиры сводятся **по egress-группам** (`group` пира, по умолчанию —
  listener-level `group`) с round-robin-балансировкой между живыми пирами этой
  группы; пир, чья несущая отвалилась, исключается при следующем выборе. Один
  слушатель — единственный порт, серверный серт и allow-list пинов — раздаёт
  столько групп, сколько объявят пиры.
- QUIC keep-alive (10 с) на обоих концах держит NAT-mapping живым со стороны
  `ss` — это тот случай, ради которого функция и существует.
