# TUN GSO / GRO / USO offload (`[tun] gso`, `[tun] gro`, `[tun] uso`)

*English version: [TUN-GSO.md](TUN-GSO.md)*

## Зачем

TUN-клиент терминирует TCP в userspace, поэтому каждый downlink-сегмент,
отдаваемый клиенту, пишется в TUN-устройство отдельным `write(2)`, а каждый
uplink-сегмент от клиента приходит отдельным `read(2)`. На нагруженном канале это
тысячи вызовов в секунду, и каждый пакет проходит весь сетевой стек ядра —
routing, `nftables`, `conntrack` и (когда клиент за WireGuard) пакетное
WG-шифрование — прежде чем попасть на NIC или в наш стек. Профилирование живого
клиента на ~40–60 Mbit показало, что CPU съедает не крипта и не userspace-стек, а
именно эта per-packet работа ядра (`nft_do_chain`, `nf_conntrack_*`, `fib_*`,
`tun_get_user`, `wg_xmit`), а `write(2)` — самый частый syscall.

Offload батчит пакеты так, что ядро и наш стек выполняются **раз на суперпакет**,
а не раз на MSS:

- **`gso` (downlink / запись, TSO):** отдаём ядру один большой суперсегмент
  (до ~60 КБ) вместо N пакетов по MSS; ядро режет его по MSS *после* того, как
  routing / `nftables` / `conntrack` / WG отработали один раз.
- **`gro` (uplink / чтение, GRO):** ядро сливает несколько входящих MSS-сегментов
  одного потока в один суперпакет >MSS (до 64 КБ) и отдаёт его целиком, так что
  наши чтение / разбор / TCP-стек выполняются раз вместо раза на MSS.

## Что делает `gso = true`

- Открывает TUN-устройство с `IFF_VNET_HDR`, поэтому каждый `read(2)` /
  `write(2)` несёт 10-байтный префикс `virtio_net_hdr`.
- **Downlink (запись):** `flush` собирает очередь server→client в один
  TCP-суперсегмент (до ~60 КБ) с `virtio_net_hdr`, задающим MSS (`gso_size`) и
  **частичную (pseudo-header) контрольную сумму**; ядро режет его на MSS-сегменты,
  проставляя каждому sequence number и финализируя L4-checksum. Здесь и появляется
  выигрыш по CPU на downlink.
- **Ретрансмит остаётся по-MSS:** scoreboard отправки трекает каждый MSS-сегмент
  отдельно, поэтому потеря внутри суперсегмента восстанавливается на гранулярности
  MSS (одиночным MSS-пакетом), ровно как без GSO.
- Сам по себе `gso` **не** запрашивает `TUNSETOFFLOAD`, поэтому путь чтения
  байт-в-байт как раньше — uplink покрывается отдельной опцией `gro`.

## Что делает `gro = true` (требует `gso`)

- Запрашивает `TUNSETOFFLOAD` с `TUN_F_CSUM | TUN_F_TSO4 | TUN_F_TSO6`, разрешая
  ядру сливать входящий **TCP** в GRO-суперпакеты (до 64 КБ, `gso_type` TCPV4/6)
  вместо предварительной нарезки до ≤MSS.
- **Uplink (чтение):** read-loop отдаёт большой TCP-сегмент напрямую в
  userspace-TCP-движок, чей приёмный путь уже принимает сегменты много больше
  одного MSS (считает границы от IP total_len, буферизует как `Bytes`, обрезает по
  окну приёма). `TUN_F_CSUM` также включает RX checksum offload, поэтому ядро
  может оставить L4-checksum недосчитанной; read-loop пересчитывает её из
  доверенного payload.
- **UDP остаётся per-datagram:** `TUN_F_USO4/6` намеренно **не** запрашивается,
  поэтому без `NETIF_F_GSO_UDP_L4` ядро ре-сегментирует UDP GRO обратно в
  датаграммы до нашего чтения. (Если UDP-суперпакет всё же придёт, read-loop
  ре-сегментирует его обратно в датаграммы, так что поведение UDP не меняется.)
- **Слияние происходит только на всплеске:** GRO сливает сегменты, только когда
  источник реально бёрстит (bulk-upload или форвардинг с
  `generic-receive-offload` на входном NIC). ACK-only / интерактивный uplink
  остаётся per-packet — это ожидаемо.
- Если ядро отвергает `TUNSETOFFLOAD`, клиент пишет предупреждение и продолжает
  без GRO; write-side TSO при этом работает.

## Что делает `uso = true` (требует `gso`)

- Запрашивает `TUNSETOFFLOAD` с `TUN_F_CSUM | TUN_F_USO4 | TUN_F_USO6`, позволяя
  writer'у отдать ядру один `GSO_UDP_L4` суперсегмент (до 64 датаграмм / ~60 КБ),
  который ядро режет на равные `gso_size` датаграммы — UDP-аналог TCP TSO из
  `gso`, на пути **записи**.
- **Downlink (запись):** per-flow reader сливает подряд идущие датаграммы одного
  размера одного потока (его 4-tuple фиксирован — все отвечают одному
  назначению) в один суперсегмент с частичной (pseudo-header) контрольной суммой,
  которую ядро финализирует на каждую датаграмму. Датаграмма другого размера
  завершает батч и переносится в следующий (zero-loss). Это выигрыш для bulk
  QUIC-видео, где downlink — тяжёлое UDP-направление.
- **`TUN_F_USO` также включает RX UDP GRO:** ядро может отдать нам слитые UDP
  суперпакеты на чтении, которые read-loop ре-сегментирует обратно в датаграммы
  (`resegment_udp_gso`), поэтому приём UDP не меняется.
- `TUN_F_CSUM` (обязателен для USO) включает RX checksum offload; read-loop
  пересчитывает недосчитанную контрольную сумму.
- Если ядро отвергает USO (< 5.18), клиент пишет предупреждение и оставляет TCP
  offload; UDP GSO отключается.

## Включение

```toml
[tun]
gso = true   # downlink TSO
gro = true   # uplink GRO (требует gso)
uso = true   # downlink UDP USO (требует gso)
```

Только Linux; на других платформах игнорируется. `gso` требует ядро с
`IFF_VNET_HDR` (есть с 2.6.27). `gro`/`uso` дополнительно требуют поддержки
`TUNSETOFFLOAD`; `uso` требует `GSO_UDP_L4` (ядро ≥ 5.18). `gro` и `uso` можно
включать независимо друг от друга и от `gso`.

> **Нюанс persistent-интерфейса.** `TUNSETOFFLOAD` ставит feature-флаги на
> устройство, переживающие рестарт процесса на *persistent* TUN-интерфейсе.
> Клиент переустанавливает offload явно при каждом attach (в т.ч. сбрасывает в
> `0`, когда выключено), поэтому обычного рестарта, как правило, достаточно.
> Если же после рестарта в `ethtool -k` видно залипшее `tx-udp-segmentation:
> off [requested on]`, продавите реальный переход флага вручную — `ethtool -K
> <tun> tx-udp-segmentation off …` (и рестарт) — либо пересоздайте интерфейс
> (`ip link del` / `ip tuntap add`).

## Проверка

Все три пути — ядро data-plane, поэтому сначала подтвердите **целостность
данных**, затем выигрыш по CPU:

1. Включите флаг(и), перезапустите клиент.
2. **Downlink (`gso`):** скачайте большой файл через туннель и сверьте
   `sha256sum` с оригиналом — байты должны совпадать.
3. **Uplink (`gro`):** загрузите большой файл через туннель и сверьте его
   `sha256sum` на дальней стороне.
4. **UDP (`uso`):** проверьте, что UDP здоров (DNS резолвится, QUIC / YouTube
   играет), и прогоните bulk-UDP; рост счётчика
   `outline_ws_tun_packets_total{outcome="uso_supersegment"}` означает, что
   downlink-датаграммы коалесцируются.
5. Сравните CPU и частоту syscall'ов с выключенным состоянием при той же
   скорости: `sudo timeout 5 strace -f -c -p $(pidof outline-ws-rust)` — с `gso`
   резко падает `write`; с `gro` на bulk-uplink — `read`.

Как читать offload через `ethtool -k <tun>`:

- **`gso` (downlink-TSO на write) в `ethtool` не виден.** Он едет на
  `virtio_net_hdr` в каждом `read`/`write` (fd открыт с `IFF_VNET_HDR`) — это
  свойство очереди, а не netdevice-фича, поэтому ни одна строка `ethtool` его не
  отражает. Единственное подтверждение — наши лог-строки `TUN loop started` /
  `TUN offload negotiated` (`gso=true`).
- **`gro` и `uso` в `ethtool` видны**, потому что ставятся через `TUNSETOFFLOAD`,
  который переключает реальные netdevice-фичи. При обоих включённых должны быть
  `on`: `tx-checksum-ip-generic` (обязательный `TUN_F_CSUM`),
  `tx-tcp-segmentation` + `tx-tcp6-segmentation` (`TUN_F_TSO4/6` от `gro`) и
  `tx-udp-segmentation` (`TUN_F_USO4/6` от `uso`). Здоровый `uso` читается как
  `tx-udp-segmentation: on` **без** хвоста `[requested on]` — форма
  `[requested on]` и есть залипшее состояние из нюанса про persistent-интерфейс
  выше.
- **`generic-segmentation-offload` / `generic-receive-offload` — НЕ наши
  флаги**: это собственные software-GSO/GRO ядра для устройства, к нашему пути
  отношения не имеют; их состояние игнорируйте.

## Откат

`gso = false` / `gro = false` / `uso = false` (значения по умолчанию) возвращают
ровно прежний per-packet путь; пересборка не нужна, только перезапуск. `gro` и
`uso` каждый убирают только свой offload, оставляя downlink-TSO, — все три
откатываются независимо.
