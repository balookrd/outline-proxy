# TUN GSO offload (`[tun] gso`)

*English version: [TUN-GSO.md](TUN-GSO.md)*

## Зачем

TUN-клиент терминирует TCP в userspace, поэтому каждый downlink-сегмент,
отдаваемый клиенту, пишется в TUN-устройство отдельным `write(2)`. На нагруженном
канале это тысячи вызовов в секунду, и каждый проходит весь сетевой стек ядра —
routing, `nftables`, `conntrack` и (когда клиент за WireGuard) пакетное
WG-шифрование — прежде чем попасть на NIC. Профилирование живого клиента на
~40–60 Mbit показало, что CPU съедает не крипта и не userspace-стек, а именно эта
per-packet работа ядра (`nft_do_chain`, `nf_conntrack_*`, `fib_*`, `tun_get_user`,
`wg_xmit`), а `write(2)` — самый частый syscall.

GSO/TSO offload на TUN-устройстве отдаёт ядру **один большой суперсегмент
(до ~60 КБ)** вместо N пакетов по MSS, так что routing / `nftables` /
`conntrack` / WG выполняются **раз на суперсегмент**, а не раз на MSS.

## Что делает `gso = true`

- Открывает TUN-устройство с `IFF_VNET_HDR`, поэтому каждый `read(2)` /
  `write(2)` несёт 10-байтный префикс `virtio_net_hdr`.
- **Downlink (запись):** `flush` собирает очередь server→client в один
  TCP-суперсегмент (до ~60 КБ) с `virtio_net_hdr`, задающим MSS (`gso_size`) и
  **частичную (pseudo-header) контрольную сумму**; ядро режет его на MSS-сегменты,
  проставляя каждому sequence number и финализируя L4-checksum. Здесь и появляется
  выигрыш по CPU.
- **Ретрансмит остаётся по-MSS:** scoreboard отправки трекает каждый MSS-сегмент
  отдельно, поэтому потеря внутри суперсегмента восстанавливается на гранулярности
  MSS (одиночным MSS-пакетом), ровно как без GSO.
- **Приём не меняется:** мы намеренно НЕ запрашиваем `TUNSETOFFLOAD` — он влияет
  только на чтение (заставил бы ядро отдавать нам GRO-суперпакеты для
  ресегментации). Без него ядро режет до ≤MSS до нашего чтения, поэтому uplink
  байт-в-байт как раньше.

## Включение

```toml
[tun]
gso = true
```

Только Linux; на других платформах игнорируется. Требуется ядро с `IFF_VNET_HDR`
(есть с 2.6.27), форвардящее GSO-суперпакеты в WireGuard (6.x — при необходимости
ядро сегментирует в софте перед WG, но per-packet стоимость routing / `nftables` /
`conntrack` уже оплачена раз на суперсегмент).

## Проверка

Путь записи — ядро data-plane, поэтому сначала подтвердите **целостность данных**,
затем выигрыш по CPU:

1. Выставьте `gso = true`, перезапустите клиент.
2. Скачайте большой файл через туннель и сверьте `sha256sum` с оригиналом — байты
   должны совпадать (неверная offload-checksum испортила бы тихо, хотя
   TCP/QUIC-проверки ниже по потоку обычно порвали бы передачу).
3. Убедитесь, что обычный сёрфинг / стриминг работает.
4. Сравните CPU и частоту `write(2)` с `gso = false` при той же скорости:
   `sudo timeout 5 strace -f -c -p $(pidof outline-ws-rust)` — число `write`
   должно резко упасть (один write на суперсегмент вместо на MSS), а `perf top` —
   показать меньше времени в сетевом стеке ядра.

## Откат

`gso = false` (значение по умолчанию) возвращает ровно прежний per-packet путь;
пересборка не нужна, только перезапуск.
