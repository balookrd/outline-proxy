# TUN GSO / GRO offload (`[tun] gso`, `[tun] gro`)

*English version: [TUN-GSO.md](TUN-GSO.md)*

## Зачем

TUN-клиент терминирует TCP в userspace, поэтому каждый downlink-сегмент,
отдаваемый клиенту, пишется в TUN-устройство отдельным `write(2)`, а каждый
uplink-сегмент от клиента приходит отдельным `read(2)`. На нагруженном канале это
тысячи вызовов в секунду, и каждый пакет проходит весь сетевой стек ядра —
routing, `nftables`, `conntrack` и (когда клиент за WireGuard) пакетное
WG-шифрование — прежде чем попасть на NIC или в наш стек. Профилирование живого
клиента на ~40–60 Mbit показало, что CPU съедает не крипта и не userspace-стек, а
именно эта per-packet работа ядра (`nft_do_chain`, `nf_conntrack_*`, `fib_*`,
`tun_get_user`, `wg_xmit`), а `write(2)` — самый частый syscall.

Offload батчит пакеты так, что ядро и наш стек выполняются **раз на суперпакет**,
а не раз на MSS:

- **`gso` (downlink / запись, TSO):** отдаём ядру один большой суперсегмент
  (до ~60 КБ) вместо N пакетов по MSS; ядро режет его по MSS *после* того, как
  routing / `nftables` / `conntrack` / WG отработали один раз.
- **`gro` (uplink / чтение, GRO):** ядро сливает несколько входящих MSS-сегментов
  одного потока в один суперпакет >MSS (до 64 КБ) и отдаёт его целиком, так что
  наши чтение / разбор / TCP-стек выполняются раз вместо раза на MSS.

## Что делает `gso = true`

- Открывает TUN-устройство с `IFF_VNET_HDR`, поэтому каждый `read(2)` /
  `write(2)` несёт 10-байтный префикс `virtio_net_hdr`.
- **Downlink (запись):** `flush` собирает очередь server→client в один
  TCP-суперсегмент (до ~60 КБ) с `virtio_net_hdr`, задающим MSS (`gso_size`) и
  **частичную (pseudo-header) контрольную сумму**; ядро режет его на MSS-сегменты,
  проставляя каждому sequence number и финализируя L4-checksum. Здесь и появляется
  выигрыш по CPU на downlink.
- **Ретрансмит остаётся по-MSS:** scoreboard отправки трекает каждый MSS-сегмент
  отдельно, поэтому потеря внутри суперсегмента восстанавливается на гранулярности
  MSS (одиночным MSS-пакетом), ровно как без GSO.
- Сам по себе `gso` **не** запрашивает `TUNSETOFFLOAD`, поэтому путь чтения
  байт-в-байт как раньше — uplink покрывается отдельной опцией `gro`.

## Что делает `gro = true` (требует `gso`)

- Запрашивает `TUNSETOFFLOAD` с `TUN_F_CSUM | TUN_F_TSO4 | TUN_F_TSO6`, разрешая
  ядру сливать входящий **TCP** в GRO-суперпакеты (до 64 КБ, `gso_type` TCPV4/6)
  вместо предварительной нарезки до ≤MSS.
- **Uplink (чтение):** read-loop отдаёт большой TCP-сегмент напрямую в
  userspace-TCP-движок, чей приёмный путь уже принимает сегменты много больше
  одного MSS (считает границы от IP total_len, буферизует как `Bytes`, обрезает по
  окну приёма). `TUN_F_CSUM` также включает RX checksum offload, поэтому ядро
  может оставить L4-checksum недосчитанной; read-loop пересчитывает её из
  доверенного payload.
- **UDP остаётся per-datagram:** `TUN_F_USO4/6` намеренно **не** запрашивается,
  поэтому без `NETIF_F_GSO_UDP_L4` ядро ре-сегментирует UDP GRO обратно в
  датаграммы до нашего чтения. (Если UDP-суперпакет всё же придёт, read-loop
  ре-сегментирует его обратно в датаграммы, так что поведение UDP не меняется.)
- **Слияние происходит только на всплеске:** GRO сливает сегменты, только когда
  источник реально бёрстит (bulk-upload или форвардинг с
  `generic-receive-offload` на входном NIC). ACK-only / интерактивный uplink
  остаётся per-packet — это ожидаемо.
- Если ядро отвергает `TUNSETOFFLOAD`, клиент пишет предупреждение и продолжает
  без GRO; write-side TSO при этом работает.

## Включение

```toml
[tun]
gso = true   # downlink TSO
gro = true   # uplink GRO (требует gso)
```

Только Linux; на других платформах игнорируется. `gso` требует ядро с
`IFF_VNET_HDR` (есть с 2.6.27). `gro` дополнительно требует поддержки
`TUNSETOFFLOAD` (давно присутствует). `gro` можно включать независимо от `gso`.

## Проверка

Оба пути — ядро data-plane, поэтому сначала подтвердите **целостность данных**,
затем выигрыш по CPU:

1. Включите флаг(и), перезапустите клиент.
2. **Downlink (`gso`):** скачайте большой файл через туннель и сверьте
   `sha256sum` с оригиналом — байты должны совпадать.
3. **Uplink (`gro`):** загрузите большой файл через туннель и сверьте его
   `sha256sum` на дальней стороне. Также проверьте, что UDP здоров (DNS
   резолвится, QUIC / YouTube играет) — UDP должен остаться per-datagram.
4. Сравните CPU и частоту syscall'ов с выключенным состоянием при той же
   скорости: `sudo timeout 5 strace -f -c -p $(pidof outline-ws-rust)` — с `gso`
   резко падает `write`; с `gro` на bulk-uplink — `read`. `ethtool -k` на
   TUN-устройстве по-прежнему покажет `tcp-segmentation-offload: off` — это
   ожидаемо: наш offload едет на `virtio_net_hdr`, а не на device-feature.

## Откат

`gso = false` / `gro = false` (значения по умолчанию) возвращают ровно прежний
per-packet путь; пересборка не нужна, только перезапуск. `gro = false` в
одиночку убирает RX GRO, оставляя downlink-TSO, — обе опции откатываются
независимо.
